Data engineers were drowning in invisible dependencies

Intuit’s Batch Processing Platform (BPP) powered thousands of data pipelines that processed financial data for products like TurboTax and QuickBooks. Engineers configured these pipelines through an Apache Airflow–based system using code and config files.

But as pipelines grew in complexity — some with 30+ processing steps and hundreds of data dependencies — a critical problem emerged: engineers had no way to visually understand how their pipeline tasks related to each other. Dependencies were buried in code, making it nearly impossible to debug failures, onboard new team members, or assess the blast radius of a single change.

Two workflows, one broken mental model

Through stakeholder interviews and contextual inquiry with 12 data engineers, I identified two core user groups with overlapping pain points:

“I have to open 4 different files and mentally trace the execution order. One wrong link and the whole pipeline fails silently at 2 AM.”

— Senior Data Engineer, TurboTax team

“When I joined, it took me 3 weeks just to understand one pipeline. There’s no picture of how data flows — just a wall of YAML.”

— Junior Data Engineer, onboarded 2 months prior

The core insight was that engineers thought about pipelines as a graph — a directed flow of processors and data assets with clear dependencies — but the tooling forced them to work in a flat, linear config format. This mismatch between mental model and tool was the root cause of slow debugging, risky deployments, and painful onboarding.

From ambiguity to a structured visual system

Phase 1 — Discover

Mapping the domain

I embedded with engineering teams for 2 weeks, shadowing pipeline authoring sessions and mapping the information architecture of Airflow DAGs. I documented every entity type (processors, data assets, time dependencies) and how they connected.

Phase 2 — Define

System modeling

I created an entity-relationship model for the DAG: nodes (processor, data asset, time dependency), edges (dependency links), and metadata (input/output counts, execution status). This became the shared language between design and engineering.

Phase 3 — Design

Canvas interaction design

I designed the node-based graph editor with drag-and-drop node placement, visual connection handles, contextual configuration panels, and zoom/pan canvas controls. Each interaction was mapped to reduce cognitive load while preserving power-user efficiency.

Phase 4 — Deliver

Component system & handoff

I built a component library in Figma covering all node states (default, hover, selected, connected, error) and all panel configurations, then worked directly with frontend engineers during implementation to ensure interaction fidelity.

A visual dependency graph that speaks the engineer’s language

The final design is a canvas-based DAG (Directed Acyclic Graph) editor embedded within the pipeline configuration experience. Engineers can visually author, connect, and configure their entire pipeline without writing dependency config manually.

Why these patterns matter

Graph-first, code-second. I designed a toggle between the visual graph and the generated code view. The graph is the primary authoring surface, but engineers can always inspect the underlying Airflow config. This dual-mode approach respects both visual thinkers and those who need code-level control.

Progressive disclosure through node expansion. Processor nodes start collapsed (showing name and type) and expand to reveal input/output data assets on demand. This prevents visual overload in complex pipelines while keeping all information one click away.

Contextual configuration, not modal interruption. Clicking a node opens a side panel rather than a modal dialog. This preserves spatial context — engineers can see where a processor sits in the graph while configuring its Spark resources, runtime engine, and time dependencies.

Error states with pipeline-level awareness. When a pipeline has validation issues, the system surfaces errors at the page level with specific guidance, and individual nodes with problems receive visual indicators. This transforms cryptic deployment failures into preventable, in-context warnings.

What I learned

This project reinforced that the best developer tools don’t abstract away complexity — they make complexity legible. The DAG editor didn’t remove the underlying Airflow configuration; it gave engineers a spatial, visual representation that matched how they already thought about their work.

Working on an internal platform also taught me the value of designing for trust. Data engineers needed to be confident that the visual tool was generating correct, production-ready configurations. I addressed this through the code view toggle, transparent error messaging, and by involving engineers deeply in the design review process.
