Visual editor for data pipelines

Designing a visual graph editor that replaced 1,000+ lines of config files with a canvas engineers could finally digest.

data pipeline internal tool enterprise

Overview

Replacing code-heavy pipeline configuration with a canvas-based DAG (Directed Acyclic Graph) editor so data engineers could see dependencies, debug faster, and onboard without a wall of config.

My role Product Designer
Company Intuit - Data Platform
Team 1 Designer, 1 PM, 8 Engineers
Tools Figma, Figma Make

Problem

Data engineers were drowning in invisible dependencies

Intuit’s Batch Processing Platform powered thousands of data pipelines for products like TurboTax and QuickBooks. As pipelines grew in complexity — some with 30+ processing steps — a set of critical problems emerged:

  • Dependencies were buried in code — no way to visually trace how pipeline tasks related to each other.
  • Debugging failures meant mentally reconstructing execution order across multiple config files.
  • New engineers faced weeks of onboarding with no visual map of data flow.

Solution

A visual dependency graph for pipeline authoring and debugging

The final design is a canvas-based DAG (Directed Acyclic Graph) editor embedded within the pipeline configuration experience. Engineers can visually author, connect, and configure their entire pipeline without writing dependency config manually.

Design rationale

Key design decisions and why

Processor-first view with expand/collapse

Pipelines can have dozens of processors, so the default view shows only processor nodes to keep dependencies clear at a glance. Users can expand any processor to reveal its input and output data assets, and collapse it back when they need the big picture.

Panel for configuring processors and data assets

Selecting a node opens a configuration panel input fields. No modal interruption — the graph stays visible for spatial context.

Empty state with guided entry

A persistent left panel provides draggable node types. The empty canvas includes instructional hints, reducing the blank-canvas anxiety for first-time users.

Design decision: When should we show the configuration?

Option A: Configure to add a node in the canvas

Pro: Focused, step-by-step configuration in a modal, preventing empty nodes on the canvas.

Con: The graph is hidden while you work, so you lose spatial context and for processor configurations, users have to go through multiple inputs and might lose progress if they close the modal.

Option B: Add first and configure later

Pro: The node stays visible so users can design the structure quickly and always see where each node sits.

Con: Nodes can sit in a draft or incomplete state until validation makes gaps obvious.

I selected Option B — to give users more flexibility on designing the pipeline structure. To prevent empty nodes being saved, I added a validation step that would be triggered after clicking 'Run' or 'Save'.

Reflection

What I learned

This project reinforced that the best developer tools don’t abstract away complexity — they make complexity legible. The DAG editor didn’t remove the underlying configuration; it gave engineers a spatial, visual representation that matched how they already thought about their work.

Working on an internal platform also taught me the value of designing for trust. Data engineers needed to be confident that the visual tool was generating correct, production-ready configurations. I addressed this through the code view toggle, transparent error messaging, and by involving engineers deeply in the design review process.
